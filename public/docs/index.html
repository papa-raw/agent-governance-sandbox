The Problem

Governance Decisions Made Blind

Multi-agent AI systems need governance, but there is no way to test which frameworks actually prevent commons collapse. Tragedy of the commons, Ostrom's principles, quadratic voting — debated in theory, never stress-tested with autonomous agents on real territorial data with real economic stakes.

Governance Models

Four Institutional Configurations

Each preset configures a complete governance stack: rules, voting, sanctions, regeneration dynamics, failure detection. Users switch between them and watch what happens.

Tragedy of the Commons
No rules. Open access. Pure self-interest. Each agent maximizes extraction without coordination.

Sponsor Integration

Storacha — Governance Memory + UCAN Enforcement

Storacha is not just a storage backend. It is the governance enforcement infrastructure.

◆

Content-Addressed Rounds

Every simulation round uploads a RoundResult JSON to a provisioned Storacha space (ags-camargue-2026). Each upload produces a real IPFS CID, gateway-verifiable at {cid}.ipfs.storacha.link.

🔒

UCAN Capability Scoping

Every agent gets an ephemeral DID (did:key:z6Mk...) and scoped capabilities. Basic agents can vote and propose. Governance bodies can also enforce and modify_rules.

☠

Adversarial Capability Testing

Chaotic agents (20%/round) and free-riders (15%/round) attempt vote spoofing, privilege escalation, ghost writes after exclusion, and history tampering. UCAN rejects every unauthorized action cryptographically.

↘

Sanction-Driven Scope Narrowing

Graduated sanctions narrow UCAN capabilities: warnings are reputation-only, penalties remove propose, suspensions make agents read-only, exclusions revoke delegation entirely.

Sponsor Integration

Lit Protocol — Threshold Disclosure Voting

Three layers of cryptography: Shamir for per-ballot privacy, Lit MPC for result integrity, Storacha for persistence. One CID proves all three.

1. Cast Vote

Approve / reject / abstain

2. Shamir Split

split(ballot, n=5, t=3)

3. Accumulate

Shares in encrypted pool

4. Threshold

Reveal when majority met

5. Tally

Simple | Super | Quadratic

6. Lit MPC Seal

litClient.encrypt() on Naga

7. Storacha

litSeal in RoundResult CID

8. Verify

litClient.decrypt() on demand

Why threshold disclosure matters

If votes are public, strategic agents adjust behavior based on partial tallies. Encrypted voting prevents this. Threshold disclosure (Braghieri, Bursztyn & Fasnacht, NBER Working Paper 34827, Feb 2026) reveals ballots only when the majority crosses a configurable threshold. This prevents strategic manipulation while preserving post-hoc accountability. The Lit seal nested inside the Storacha CID means one content address proves both integrity and persistence.

Quick Start

# Install and run npm install npm run dev # localhost:5173 # Build for production npm run build

Environment Variables

VITE_STORACHA_KEY — Stable ed25519 key for Storacha BYOD pattern
VITE_STORACHA_PROOF — Base64 delegation proof for Storacha space
VITE_LIT_ENABLED — Set false to disable Lit Protocol (default: true)
VITE_UCAN_ENABLED — Set false to disable UCAN enforcement (default: true)
VITE_ANTHROPIC_API_KEY — Enable Claude-powered agent decisions

Without any env vars, the app runs with deterministic agents, local pseudo-CIDs, and Shamir-only voting. Each integration activates independently when its credentials are provided.
